### C: 실행 시작과 기본 출력 담당

경험이 적은 사람이 맡되, `argument_passing`은 초반 난관이므로 A가 설계와 리뷰를 같이 한다.

## 전체 흐름

C 담당 작업은 크게 두 흐름으로 나뉜다.

| 흐름 | 핵심 함수 | 목표 |
|---|---|---|
| 프로그램 시작 흐름 | `process_create_initd`, `initd`, `process_exec`, `load`, `setup_stack`, `_start` | command line을 `argc`/`argv`로 바꾸고 사용자 프로그램 시작 시점에 전달한다. |
| 기본 syscall 흐름 | `halt`, `exit`, `write`, `syscall_entry`, `syscall_handler`, `process_exit` | 사용자 프로그램이 커널에 요청한 halt/exit/write를 처리하고, 테스트 출력과 종료 메시지를 안정화한다. |

첫 번째 흐름이 깨지면 `args-*` 테스트에서 `argc`, `argv[i]`, `argv[argc] == NULL`, stack alignment가 틀어진다.

두 번째 흐름이 깨지면 테스트의 `msg()` 출력 자체가 보이지 않거나, `exit: exit(57)` 같은 종료 메시지가 맞지 않거나, `halt()` 이후에도 프로그램이 계속 실행된다.

## Argument Passing 관련 함수

### `process_create_initd`

위치: `pintos/userprog/process.c`

역할:

- Pintos가 처음 실행할 사용자 프로그램을 만들 때 호출되는 함수다.
- 입력으로 받은 command line 문자열을 페이지 단위 버퍼에 복사한다.
- 새 커널 스레드를 만들고, 복사한 문자열을 `initd`에 넘긴다.
- 주석에도 적혀 있듯이, 원본 문자열을 그대로 공유하면 호출자와 loader 사이에 race가 생길 수 있으므로 복사본을 만든다.

C 담당 관점에서 중요한 점:

- 이 함수가 넘기는 문자열은 단순 파일명일 수도 있고, `"args-single onearg"`처럼 인자가 붙은 command line일 수도 있다.
- 복사된 문자열은 이후 `process_exec`와 `load` 쪽에서 실행 파일 이름과 인자 목록으로 나뉘어야 한다.
- thread 이름과 실제 실행 파일 이름을 어떻게 다룰지는 팀 공통 정책과 맞춰야 한다. 다만 파일 시스템에서 실행 파일을 열 때는 전체 command line이 아니라 첫 번째 토큰이 필요하다.

완료 기준:

- command line 복사본의 lifetime이 `process_exec`가 사용할 때까지 유지된다.
- 실패 시 할당한 페이지를 잃어버리지 않는다.
- `args-*` 실행 시 첫 번째 토큰이 `argv[0]`가 될 수 있는 정보가 사라지지 않는다.

### `initd`

위치: `pintos/userprog/process.c`

역할:

- 첫 사용자 프로세스를 시작하는 커널 스레드 함수다.
- 필요한 초기화를 한 뒤 `process_exec`를 호출해서 실제 사용자 프로그램으로 전환한다.
- `process_exec`가 실패하면 첫 사용자 프로그램을 띄우지 못한 것이므로 panic 경로로 간다.

C 담당 관점에서 중요한 점:

- `initd` 자체가 argument passing을 직접 구현하는 곳은 아니다.
- 하지만 `initd`가 `process_exec`에 넘기는 값이 command line 전체이므로, 인자 전달 흐름의 시작점으로 봐야 한다.
- `process_exec`가 성공하면 이 함수는 다시 돌아오지 않는다. 사용자 프로그램으로 CPU 상태가 전환되기 때문이다.

완료 기준:

- `process_exec`에 command line이 정상 전달된다.
- 실패 경로에서 할당 자원이 새지 않도록 `process_exec` 쪽 정리와 충돌하지 않는다.

### `process_exec`

위치: `pintos/userprog/process.c`

역할:

- 현재 커널 스레드의 실행 컨텍스트를 새로운 사용자 프로그램으로 바꾸는 함수다.
- 새 사용자 프로그램이 사용할 `intr_frame`을 준비한다.
- 기존 사용자 주소 공간을 정리한 뒤, ELF loader인 `load`를 호출한다.
- `load`가 성공하면 `do_iret`을 통해 사용자 모드로 넘어간다.

Argument passing에서 하는 일:

- 입력으로 받은 command line을 분석할 준비를 한다.
- 실행 파일을 열 때 필요한 파일명과 사용자 프로그램에 넘길 인자 목록을 구분해야 한다.
- `"grep foo bar"`가 들어왔다면, 실행 파일을 찾을 때 필요한 이름은 `grep`이고, 사용자 프로그램이 받는 값은 `argc == 3`, `argv[0] == "grep"`, `argv[1] == "foo"`, `argv[2] == "bar"` 형태다.
- 이 함수에서 파싱을 끝낼지, `load`로 command line 정보를 넘길지는 팀 구현 방식에 따라 달라질 수 있다. 중요한 것은 `load`가 파일을 열 때 전체 command line을 파일명으로 쓰면 안 된다는 점이다.

연결되는 함수:

- `process_cleanup`: 기존 사용자 주소 공간을 제거한다.
- `load`: ELF 파일을 로드하고 초기 스택과 레지스터를 준비한다.
- `do_iret`: 준비된 `intr_frame`으로 사용자 모드에 진입한다.

주의점:

- 현재 skeleton은 `load(file_name, &_if)` 형태라서, `file_name`이 `"args-single onearg"`이면 그대로 파일을 열려고 할 위험이 있다.
- `process_exec`는 인자로 받은 페이지를 마지막에 해제한다. 파싱 중 추가 복사본을 만들면 그 복사본의 해제 책임도 분명해야 한다.
- `process_exec`가 성공하면 반환하지 않으므로, 성공 이후에 해야 할 정리 작업을 기대하면 안 된다.

완료 기준:

- 실행 파일 이름과 인자 목록이 분리된다.
- `load` 실패 시 `-1`을 반환한다.
- `load` 성공 시 사용자 프로그램의 `_start(argc, argv)`가 올바른 값으로 시작한다.

### `load`

위치: `pintos/userprog/process.c`

역할:

- ELF 실행 파일을 현재 스레드의 사용자 주소 공간에 로드한다.
- 페이지 테이블을 만들고 활성화한다.
- 실행 파일을 열고 ELF header와 program header를 검증한다.
- load 가능한 segment를 사용자 주소 공간에 매핑한다.
- `setup_stack`으로 초기 사용자 스택을 만든다.
- 사용자 프로그램의 시작 주소를 `intr_frame`의 `rip`에 넣는다.

Argument passing에서 하는 일:

- `setup_stack` 이후, 사용자 스택 위에 인자 문자열과 `argv` 포인터 배열을 배치해야 한다.
- `intr_frame`의 `rsp`는 최종 사용자 스택 포인터를 가리켜야 한다.
- x86-64 규칙에 따라 `_start`의 첫 번째 인자인 `argc`는 `RDI`, 두 번째 인자인 `argv`는 `RSI`에 들어가야 한다.
- `argv`에 넣는 값은 첫 번째 문자열 주소가 아니라, `argv[0]` 포인터가 저장된 배열의 시작 주소다.

연결되는 함수:

- `filesys_open`: 실행 파일을 연다.
- `validate_segment`: ELF segment가 유효한지 검사한다.
- `load_segment`: segment를 실제 사용자 메모리에 올린다.
- `setup_stack`: 사용자 스택 페이지를 만든다.

주의점:

- `load`에서 파일을 열 때 넘기는 이름은 실행 파일 이름이어야 한다.
- `load`가 끝날 때 실행 파일을 닫는 skeleton 흐름이 있다. 이후 `rox-*`를 맡는 담당자는 executable write deny 때문에 이 부분을 다시 볼 수 있다. C 담당은 초반 `args-*`, `halt`, `exit`, `write(fd=1)` 안정화에 집중한다.
- 스택에 넣는 문자열은 사용자 주소 공간에 있어야 한다. 커널 버퍼의 주소를 `argv`에 넣으면 사용자 프로그램이 사용할 수 없다.

완료 기준:

- `if_->rip`이 사용자 프로그램 entry point를 가리킨다.
- `if_->rsp`가 argument data까지 반영된 최종 스택 위치를 가리킨다.
- `if_->R.rdi`가 `argc`, `if_->R.rsi`가 `argv` 배열 시작 주소를 담는다.
- `args-none`, `args-single`, `args-multiple`, `args-many`, `args-dbl-space`가 기대하는 `argc`와 `argv` 구조를 만들 수 있다.

### `setup_stack`

위치: `pintos/userprog/process.c`

역할:

- 사용자 스택으로 쓸 페이지 하나를 할당한다.
- 해당 페이지를 `USER_STACK - PGSIZE` 위치에 매핑한다.
- 초기 `rsp`를 `USER_STACK`으로 설정한다.

Argument passing에서 하는 일:

- `setup_stack` 자체는 기본 스택 페이지만 만든다.
- 실제 인자 문자열, padding, `argv[argc] == NULL`, `argv` 포인터 배열, fake return address는 `setup_stack` 이후 `rsp`를 아래로 내리면서 배치한다.
- `args.c`는 `argv` 주소가 8바이트 정렬인지 검사한다. 따라서 최종 배치에서 alignment가 중요하다.

스택에 들어가야 하는 값:

- 각 인자 문자열과 끝의 null 문자
- 각 문자열의 시작 주소
- `argv[argc]`에 해당하는 null pointer
- `argv[0]`부터 `argv[argc - 1]`까지의 포인터 배열
- fake return address

주의점:

- 스택은 높은 주소에서 낮은 주소로 자란다.
- 문자열을 먼저 넣고, 그 문자열들의 주소를 따로 기억해 두어야 나중에 `argv` 배열을 만들 수 있다.
- `RSI`에는 문자열 시작 주소가 아니라 `argv[0]` 포인터가 놓인 주소를 넣어야 한다.
- `RDI`에는 인자 개수인 `argc`를 넣어야 한다.

완료 기준:

- 사용자 프로그램의 `_start`가 `argc`, `argv`를 정상적으로 받는다.
- `argv[argc]`가 null이다.
- `argv` 배열이 8바이트 정렬되어 있다.
- 여러 공백이 있는 command line도 빈 인자를 만들지 않는다.

### `_start`

위치: `pintos/lib/user/entry.c`

역할:

- 모든 사용자 프로그램의 실제 시작 함수다.
- 커널이 준비해 둔 `argc`, `argv`를 받아서 `main(argc, argv)`를 호출한다.
- `main`이 반환한 값을 `exit` syscall로 넘긴다.

C 담당 관점에서 중요한 점:

- 커널이 직접 `main`을 호출하는 것이 아니라 `_start`가 먼저 실행된다.
- 따라서 argument passing의 최종 수신자는 `_start`다.
- `_start`가 받는 첫 번째 인자와 두 번째 인자는 x86-64 calling convention에 따라 각각 `RDI`, `RSI`에서 온다.

완료 기준:

- `main`이 기대하는 `argc`, `argv`가 그대로 들어간다.
- `main`이 `return 0`을 하면 최종적으로 `exit(0)`이 호출되어 종료 메시지에 반영된다.

### `args.c` 테스트 프로그램

위치: `pintos/tests/userprog/args.c`

역할:

- C 담당의 argument passing 결과를 관찰하는 테스트 프로그램이다.
- `argc`를 출력한다.
- `argv[0]`부터 `argv[argc]`까지 출력한다.
- `argv[argc]`가 null인지 확인할 수 있게 출력한다.
- `argv`와 stack alignment가 8바이트 기준에 맞는지도 확인한다.

테스트가 보는 것:

- `args-none`: 프로그램 이름만 있는 경우 `argc == 1`
- `args-single`: 인자 하나가 추가된 경우 `argc == 2`
- `args-multiple`: 여러 인자 순서가 유지되는지 확인
- `args-dbl-space`: 여러 공백이 빈 인자로 들어가지 않는지 확인
- `args-many`: 많은 인자를 넣어도 스택 구성이 깨지지 않는지 확인

주의점:

- 테스트 출력은 `msg()`를 통해 나오고, `msg()`는 내부적으로 `write(STDOUT_FILENO, ...)`를 호출한다.
- 따라서 argument passing이 맞아도 `write(fd=1)`이 없으면 출력 확인이 어렵다.
- main process가 child를 기다리는 최소 `process_wait`가 없으면 출력이 끝나기 전에 Pintos가 종료될 수 있다. 이 부분은 B 담당과 맞춰야 한다.

## 기본 syscall 관련 함수

### 사용자 영역 `halt`

위치: `pintos/lib/user/syscall.c`

역할:

- 사용자 프로그램이 호출하는 wrapper 함수다.
- `SYS_HALT` 번호로 커널에 syscall을 요청한다.
- 정상적으로 동작하면 반환하지 않는다.

C 담당 관점에서 중요한 점:

- 이 함수 자체는 이미 사용자 라이브러리에 있다.
- C 담당이 주로 봐야 하는 곳은 커널 쪽 syscall 처리다.
- `halt.c` 테스트는 `halt()` 이후의 `fail()`이 실행되면 안 된다고 본다.

완료 기준:

- `halt()`를 호출하면 Pintos가 종료된다.
- 테스트 출력에 `(halt) begin`은 있고 `(halt) fail`은 없어야 한다.

### 사용자 영역 `exit`

위치: `pintos/lib/user/syscall.c`

역할:

- 사용자 프로그램이 종료 상태를 커널에 넘기는 wrapper 함수다.
- `SYS_EXIT` 번호와 status 인자를 syscall로 전달한다.
- 정상적으로 동작하면 반환하지 않는다.

C 담당 관점에서 중요한 점:

- `exit(57)`은 커널에 숫자 57을 전달해야 한다.
- 커널은 이 값을 현재 프로세스의 종료 상태로 기록하고, 종료 메시지와 `wait` 결과에 연결해야 한다.
- `wait` 결과는 B 담당 범위지만, status 저장 위치와 의미는 C/B가 반드시 맞춰야 한다.

완료 기준:

- `exit.c` 테스트에서 `exit: exit(57)`이 출력된다.
- `exit()` 뒤의 `fail()`은 실행되지 않는다.

### 사용자 영역 `write`

위치: `pintos/lib/user/syscall.c`

역할:

- 사용자 프로그램이 파일 또는 콘솔에 데이터를 쓰기 위해 호출하는 wrapper 함수다.
- `SYS_WRITE` 번호와 `fd`, `buffer`, `size` 인자를 syscall로 전달한다.
- 반환값은 실제로 쓴 byte 수여야 한다.

C 담당 관점에서 중요한 점:

- C 담당의 초반 범위는 `write(fd=1)`, 즉 stdout 출력이다.
- `tests/lib.c`의 `msg()`는 `STDOUT_FILENO`로 메시지를 한 번에 출력한다.
- `printf`, `puts`, `putchar`, 테스트의 `msg()`는 결국 `write(1, ...)`에 의존한다.
- 파일 fd에 대한 `write`는 D 담당의 fd table/file syscall 범위다. 다만 같은 syscall 번호를 쓰므로 dispatch와 pointer 검증 정책은 A/D와 맞춰야 한다.

완료 기준:

- `write(1, buffer, size)`가 콘솔에 정확히 `size` byte를 출력한다.
- 반환값은 출력한 byte 수다.
- `size == 0`이면 아무것도 출력하지 않고 0을 반환해야 한다.
- 사용자 buffer가 잘못된 주소이면 kernel panic이 아니라 해당 프로세스 종료 또는 합의된 error path로 가야 한다. 이 검증 helper는 A 담당과 맞춘다.

### `syscall_init`

위치: `pintos/userprog/syscall.c`

역할:

- CPU의 syscall 진입 지점을 설정한다.
- `syscall` 명령이 실행되면 어디로 점프할지 MSR에 등록한다.
- 현재 skeleton에서는 `syscall_entry`가 진입점으로 설정된다.

C 담당 관점에서 중요한 점:

- 보통 C 담당이 직접 많이 바꿀 함수는 아니다.
- 하지만 사용자 wrapper에서 `syscall` 명령을 실행하면 이 설정을 통해 커널의 `syscall_entry`로 들어온다는 흐름은 알아야 한다.

완료 기준:

- 사용자 syscall이 커널 `syscall_entry`로 진입한다.

### `syscall_entry`

위치: `pintos/userprog/syscall-entry.S`

역할:

- assembly로 작성된 syscall 진입부다.
- 사용자 레지스터 상태를 커널 스택에 저장해서 `struct intr_frame` 형태로 만든다.
- 커널 스택으로 전환한 뒤 `syscall_handler`를 호출한다.
- `syscall_handler`가 끝나면 저장해 둔 사용자 상태를 복원하고 `sysretq`로 사용자 모드에 돌아간다.

C 담당 관점에서 중요한 점:

- 사용자 영역 syscall wrapper는 syscall 번호를 `rax`에, 인자를 `rdi`, `rsi`, `rdx`, `r10`, `r8`, `r9` 순서로 넣는다.
- `syscall_handler`에서는 이 값들이 `intr_frame` 안에 저장된 상태로 보인다.
- 반환값이 있는 syscall은 `intr_frame`의 `rax` 위치에 값을 넣어야 사용자 wrapper가 그 값을 반환받는다.

완료 기준:

- C의 `halt`, `exit`, `write` 요청이 `syscall_handler`에서 구분 가능한 register 상태로 들어온다.

### `syscall_handler`

위치: `pintos/userprog/syscall.c`

역할:

- 모든 syscall 요청을 받는 커널 쪽 중심 함수다.
- 현재 skeleton은 `"system call!"`을 출력하고 스레드를 종료하는 임시 상태다.
- 실제 구현에서는 syscall 번호를 읽고, 번호별 처리 함수 또는 분기 로직으로 보내야 한다.

C 담당 관점에서 필요한 syscall:

- `SYS_HALT`: Pintos 종료
- `SYS_EXIT`: 현재 프로세스 종료
- `SYS_WRITE`: 특히 `fd == 1`인 stdout 출력

인자 위치:

| syscall | syscall 번호 | 1번 인자 | 2번 인자 | 3번 인자 | 반환값 |
|---|---|---|---|---|---|
| `halt` | `rax` | 없음 | 없음 | 없음 | 반환하지 않음 |
| `exit` | `rax` | `rdi` = status | 없음 | 없음 | 반환하지 않음 |
| `write` | `rax` | `rdi` = fd | `rsi` = buffer | `rdx` = size | `rax` = 쓴 byte 수 |

주의점:

- dispatch 구조 자체는 A 담당이 잡는 범위로 되어 있다. C 담당은 A가 정한 구조에 맞춰 `halt`, `exit`, `write(fd=1)` 동작을 연결해야 한다.
- 알 수 없는 syscall 번호나 잘못된 인자는 kernel panic으로 이어지지 않도록 팀 정책을 맞춰야 한다.
- `write`의 buffer는 사용자 주소다. 커널이 그대로 신뢰하면 안 되며, A 담당의 user pointer 검증 정책을 따라야 한다.

완료 기준:

- `halt`, `exit`, `write(fd=1)`가 syscall 번호별로 구분된다.
- `write(fd=1)`는 출력 byte 수를 반환한다.
- syscall 처리 중 불필요한 debug 출력이 남아 있지 않다. 테스트 expected output은 추가 출력에 민감하다.

### kernel 쪽 `halt` 처리

위치 후보: `pintos/userprog/syscall.c`

관련 함수: `power_off` in `pintos/threads/init.c`

역할:

- 사용자 프로그램이 `halt()`를 호출했을 때 Pintos 자체를 종료한다.
- 일반 프로세스 종료가 아니라 전체 머신 종료에 가까운 동작이다.

테스트 근거:

- `pintos/tests/userprog/halt.c`는 `halt()` 호출 뒤 `fail()`이 실행되면 안 된다고 본다.
- `pintos/tests/userprog/halt.ck`는 `(halt) begin`이 출력되고 `(halt) fail`이 없어야 한다고 확인한다.

주의점:

- `halt`는 현재 프로세스만 종료하는 `exit`와 다르다.
- 반환해서 사용자 프로그램으로 돌아가면 안 된다.
- 종료 과정에서 파일 시스템 정리 등은 `power_off` 쪽에서 처리되는 흐름을 가진다.

완료 기준:

- `halt()` 호출 후 사용자 프로그램의 다음 줄이 실행되지 않는다.
- 테스트 output에 불필요한 커널 debug 메시지가 끼지 않는다.

### kernel 쪽 `exit` 처리

위치 후보: `pintos/userprog/syscall.c`, `pintos/userprog/process.c`

관련 함수: `thread_exit`, `process_exit`

역할:

- 사용자 프로그램이 넘긴 종료 status를 현재 프로세스의 종료 상태로 기록한다.
- 현재 프로세스를 종료한다.
- 종료 시 정해진 형식의 종료 메시지를 출력한다.
- 이후 B 담당의 `wait`가 이 status를 회수할 수 있어야 한다.

테스트 근거:

- `pintos/tests/userprog/exit.c`는 `exit(57)` 호출 뒤 `fail()`이 실행되면 안 된다고 본다.
- `pintos/tests/userprog/exit.ck`는 `exit: exit(57)` 출력이 있어야 한다고 본다.
- `args-*` 테스트도 `main`이 0을 반환하면 `_start`를 거쳐 `exit(0)`으로 종료되므로 `args-none: exit(0)` 같은 메시지를 기대한다.

주의점:

- 종료 메시지는 테스트 output과 직접 비교된다. 형식, 공백, 추가 출력이 중요하다.
- process 이름은 테스트 expected output의 앞부분과 맞아야 한다. 예를 들어 `exit` 테스트는 `exit: exit(57)`을 기대한다.
- status 저장은 B 담당의 `process_wait`/child record 설계와 연결된다.
- page fault나 invalid user pointer 같은 비정상 종료는 보통 `-1` status와 연결된다. 이 부분은 A/B 담당과 정책을 맞춘다.

완료 기준:

- 정상 `exit(status)`가 해당 status로 종료된다.
- 종료 메시지가 정확히 한 번 출력된다.
- `exit()` 뒤의 사용자 코드가 실행되지 않는다.
- 나중에 `wait`가 붙었을 때 부모가 같은 status를 받을 수 있는 구조와 충돌하지 않는다.

### `process_exit`

위치: `pintos/userprog/process.c`

역할:

- `thread_exit`에서 호출되는 프로세스 종료 정리 함수다.
- 현재 skeleton에는 종료 메시지와 리소스 정리 TODO가 있다.
- 마지막에는 `process_cleanup`을 통해 사용자 주소 공간을 정리한다.

C 담당 관점에서 중요한 점:

- `exit` syscall 자체는 `syscall_handler`에서 들어오지만, 종료 메시지를 어디에서 출력할지는 `process_exit`와 연결된다.
- `thread_exit`를 통해 종료되는 모든 사용자 프로세스가 `process_exit`를 거치므로, 정상 종료와 비정상 종료의 status 값을 어떻게 보관할지 미리 정해야 한다.
- cleanup 전체는 B 담당의 lifecycle과도 연결되지만, C 담당은 최소한 `exit` 테스트와 `args-*`의 `exit(0)` 출력이 맞게 되도록 협업해야 한다.

완료 기준:

- `process_cleanup`이 정상 호출된다.
- 종료 메시지는 테스트 기대 형식과 맞는다.
- status가 초기화되지 않은 값으로 출력되지 않는다.

### kernel 쪽 `write(fd=1)` 처리

위치 후보: `pintos/userprog/syscall.c`

관련 함수: kernel console 출력 경로, `putbuf`

역할:

- 사용자 프로그램의 stdout 출력을 커널 콘솔로 보낸다.
- fd `1`은 `STDOUT_FILENO`이고, 로컬 `pintos/include/lib/stdio.h`에 정의되어 있다.
- 테스트의 `msg()`, 사용자 `printf`, `puts`, `putchar`는 결국 `write(1, ...)`를 사용한다.

테스트 근거:

- `pintos/tests/lib.c`의 `msg()`는 메시지를 buffer 하나로 만든 뒤 `write(STDOUT_FILENO, buf, strlen(buf))`를 호출한다.
- `args-*`, `halt`, `exit`를 포함한 많은 테스트가 `msg()` 출력에 의존한다.
- `write-zero`는 `size == 0`일 때 0을 반환해야 한다고 본다.
- `write-stdin`은 fd `0`에 쓰는 경우 실패하거나 `-1` 종료가 가능하다고 설명한다.
- `write-bad-fd`는 잘못된 fd에 대한 처리를 본다. 이 부분은 D 담당 fd table과 연결된다.
- `write-bad-ptr`, `write-boundary`는 buffer pointer 검증과 page boundary 처리를 본다. 이 부분은 A 담당 user memory helper와 연결된다.

주의점:

- C 담당의 초반 목표는 fd `1` stdout이다.
- 일반 파일 fd write는 D 담당의 파일 syscall과 fd table이 필요하다.
- buffer는 사용자 주소이므로, 출력 전에 user memory 검증이 필요하다.
- 여러 byte를 출력할 때 byte마다 따로 출력하면 출력이 섞이기 쉬우므로, 테스트 라이브러리는 한 번의 `write`로 한 메시지를 내보내도록 되어 있다.
- 반환값은 실제 출력한 byte 수여야 한다. stdout에서는 정상 처리 시 `size`가 된다.
- `size == 0`은 정상적인 0 byte write로 봐야 한다.

완료 기준:

- `msg()` 출력이 expected output과 같은 순서와 내용으로 보인다.
- `write(1, buffer, size)`가 정상 buffer에 대해 `size`를 반환한다.
- stdout 출력 때문에 kernel panic이 나지 않는다.
- 파일 fd 처리와 충돌하지 않도록 D 담당 fd 정책과 분리되어 있다.

## C 담당 체크리스트

### `argument_passing`

- command line을 실행 파일 이름과 인자 목록으로 분리했는지 확인한다.
- 여러 공백이 빈 인자로 들어가지 않는지 확인한다.
- `argv[0]`이 실행 파일 이름과 일치하는지 확인한다.
- 문자열은 사용자 스택에 저장했는지 확인한다.
- `argv` 배열에는 사용자 스택 안 문자열들의 주소가 들어가는지 확인한다.
- `argv[argc]`가 null인지 확인한다.
- `argv` 배열 시작 주소가 `RSI`에 들어가는지 확인한다.
- `argc`가 `RDI`에 들어가는지 확인한다.
- fake return address가 들어가는지 확인한다.
- stack/argv alignment가 `args.c`의 검사에 걸리지 않는지 확인한다.

### `halt`

- `SYS_HALT`가 다른 syscall과 구분되는지 확인한다.
- 호출 후 사용자 프로그램으로 돌아오지 않는지 확인한다.
- `(halt) fail` 같은 후속 출력이 나오지 않는지 확인한다.

### `exit`

- `SYS_EXIT`의 status 인자를 올바른 register 위치에서 읽는지 확인한다.
- status가 현재 프로세스 종료 상태로 보관되는지 확인한다.
- 종료 메시지 형식이 `프로세스이름: exit(status)`인지 확인한다.
- `exit()` 뒤의 사용자 코드가 실행되지 않는지 확인한다.
- `main`의 반환값이 `_start`를 거쳐 `exit(return_value)`로 이어지는지 확인한다.

### `write(fd=1)`

- `SYS_WRITE`에서 fd, buffer, size를 올바른 register 위치에서 읽는지 확인한다.
- fd `1`을 stdout으로 처리하는지 확인한다.
- `size == 0`에서 0을 반환하는지 확인한다.
- stdout 정상 출력의 반환값이 출력 byte 수인지 확인한다.
- 사용자 buffer 검증 정책이 A 담당 helper와 일관되는지 확인한다.
- fd `0`, 잘못된 fd, 일반 파일 fd 처리는 D 담당 정책과 충돌하지 않는지 확인한다.

## 핵심 대기 관계

| 기다리는 작업 | 막히는 작업 | 이유 |
|---|---|---|
| A의 syscall dispatch | C의 `halt/exit/write`, D의 file syscall, B의 syscall 형태 `exec/wait/fork` | syscall handler가 분기하지 못하면 기능 handler를 호출할 수 없다. |
| C의 `write(fd=1)`와 B의 최소 `process_wait` | `args-*`의 안정적인 확인 | user program이 출력하고, main process가 child를 기다려야 expected output을 볼 수 있다. |
