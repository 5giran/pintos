# WIL: Pintos Project 3 Virtual Memory 학습 정리

> 기간: 2026.05.08 ~ 2026.05.10  
> 참고 문서: `docs/2026.05.08.md`, `docs/2026.05.09.md`, `docs/2026.05.10.md`, `docs/ai/pintos_project3_virtual_memory_appendix_ko (1).md`

## 한 줄 요약

이번 기간에는 Pintos Project 3의 Virtual Memory가 단순히 페이지를 메모리에 올리는 기능이 아니라, **가상 주소를 기준으로 필요한 시점에 page를 준비하고, 부족한 physical frame을 관리하며, 파일과 swap disk까지 연결하는 전체 메모리 관리 흐름**이라는 점을 배웠다.

## 2026.05.08에 배운 것

가상 메모리의 큰 그림을 잡았다. 기존에는 프로그램 실행에 필요한 내용을 메모리에 바로 올려야 한다고 생각하기 쉬운데, VM에서는 모든 내용을 즉시 올리지 않고 필요한 정보만 기록해 둔 뒤 실제 접근 시 page fault를 통해 데이터를 가져온다는 점을 정리했다.

주요 용어도 구분했다.

- `Page`: 가상 메모리 영역이며 Pintos에서는 4KB 단위다.
- `Frame`: 실제 물리 메모리 영역이며 page와 같은 4KB 단위다.
- `Page table`: 가상 page와 physical frame의 매핑을 저장한다.
- `Swap slot`: 물리 메모리가 부족할 때 frame 내용을 디스크에 임시 저장하는 공간이다.
- `Supplemental Page Table, SPT`: page fault가 났을 때 해당 가상 page가 어떤 상태인지, 어디서 로드해야 하는지, writable인지 등을 판단하기 위한 보조 테이블이다.

또한 `page-aligned`가 주소의 하위 12비트가 0인 4KB 경계 정렬이라는 점을 이해했다. 예를 들어 가상 주소의 하위 부분은 페이지 내부 offset으로 쓰이고, SPT에서는 보통 `pg_round_down()`으로 page boundary에 맞춘 주소를 key로 사용해야 한다.

`anonymous page`와 `file-backed page`의 차이도 배웠다. anonymous page는 stack, heap, zero-filled page처럼 원본 파일이 없는 메모리이고, file-backed page는 파일을 원본 저장소로 갖는 page다.

## 2026.05.09에 배운 것

VM 구현에 필요한 핵심 자료구조를 더 구체적으로 정리했다.

첫 번째는 `SPT`다. SPT는 각 process가 자기 가상 주소 공간에 대해 별도로 갖는 자료구조이며, 보통 `struct hash pages` 형태로 구현한다. key는 page-aligned virtual address이고, value는 `struct page`다. `struct page` 안에는 `va`, `frame`, `operations`, `writable`, `hash_elem`, 그리고 `uninit_page`, `anon_page`, `file_page` 같은 타입별 정보가 들어간다.

두 번째는 `Frame Table`이다. 사용 가능한 frame이 없으면 eviction을 수행해야 하며, 이때 어떤 frame을 내보낼지 선택하고 page와 frame의 연결을 갱신해야 한다.

세 번째는 `Swap Table`이다. anonymous page가 eviction될 때 디스크의 swap slot에 저장되고, 다시 필요해지면 swap slot에서 읽어 새 frame에 복구한다.

`Stack growth`도 page fault와 연결해서 이해했다. stack이 더 필요할 때 page fault를 계기로 새 anonymous page를 추가해 stack 영역을 확장하는 기능이다.

`dirty bit`에 대해서는 더 정확히 이해했다. dirty bit는 page가 수정되었는지 알려주지만, page 안의 어느 byte 또는 어느 문자열 부분이 바뀌었는지까지 알려주는 것은 아니다. 따라서 file-backed page나 mmap page를 처리할 때는 dirty 여부를 기준으로 page 단위 write-back을 판단해야 한다.

`Memory-mapped file`도 정리했다. `mmap`은 파일 내용을 가상 메모리 주소 범위에 연결하고, 실제 접근 시 필요한 page를 읽어 오게 한다. 이 기능은 lazy loading, dirty bit, eviction, munmap write-back이 함께 맞물려야 한다.

## 2026.05.10에 배운 것

기존 Project 2의 `load_segment()` 방식과 Project 3 VM 방식의 차이를 이해했다. 기존에는 `load_segment()`에서 파일 내용을 page 단위로 바로 메모리에 읽어 올 수 있었지만, VM에서는 실제 메모리에 즉시 올리지 않고 page fault 시점에 로드할 수 있도록 정보를 SPT에 등록해야 한다.

따라서 `load_segment()` 내부에서는 page 단위로 반복하면서 `vm_alloc_page_with_initializer()`를 호출하고, 나중에 page fault가 발생했을 때 실제 파일에서 데이터를 읽을 수 있도록 필요한 정보를 page에 넘겨야 한다. 이때 필요한 정보는 파일 객체, offset, read_bytes, zero_bytes, writable 같은 값들이다.

`uninit page`의 역할도 배웠다. uninit page는 아직 실제 타입이 완전히 초기화되지 않은 lazy loading용 page이며, 처음 접근할 때 initializer를 통해 anonymous page 또는 file-backed page로 구체화된다.

또한 `user virtual address`가 해당 process의 user address space 안에서 프로그램이 사용하는 주소라는 점을 정리했다. 커널 주소와 사용자 주소를 구분해야 page fault 처리, pointer validation, SPT lookup에서 잘못된 접근을 막을 수 있다.

마지막으로 Pintos의 hash table 구조를 학습했다. Pintos hash table은 chaining 방식이며, hash table이 원소 노드를 직접 할당하지 않는다. 대신 사용자가 만든 구조체 안에 `struct hash_elem`을 넣고, hash table API는 그 `hash_elem`을 기준으로 동작한다.

`struct hash`의 주요 구성은 다음과 같다.

- `elem_cnt`: 현재 들어 있는 원소 수
- `bucket_cnt`: bucket 개수이며 항상 2의 거듭제곱
- `buckets`: `struct list` 배열
- `hash`: 원소의 hash 값을 계산하는 callback
- `less`: 두 원소를 비교하는 callback
- `aux`: `hash`, `less`에 함께 넘기는 보조 데이터

SPT를 hash table로 구현하려면 `struct page` 안에 `hash_elem`을 넣고, `va`를 기준으로 hash function과 less function을 작성해야 한다. 검색할 때는 fault address를 page boundary로 내린 뒤 그 주소를 key로 page를 찾는 흐름이 된다.

## 내가 이해한 VM 전체 흐름

이번 학습을 통해 Project 3의 흐름을 다음처럼 연결해서 이해하게 되었다.

1. process가 실행되면 실행 파일의 segment 정보를 읽는다.
2. VM에서는 segment 내용을 즉시 frame에 올리지 않고, SPT에 lazy page 정보를 등록한다.
3. user program이 해당 virtual address에 접근하면 page fault가 발생한다.
4. page fault handler는 fault address를 기준으로 SPT를 조회한다.
5. 유효한 page라면 frame을 확보한다.
6. frame이 부족하면 frame table에서 victim을 골라 eviction한다.
7. page type에 따라 file에서 읽거나, swap에서 복구하거나, zero page를 만든다.
8. page와 frame을 연결하고 hardware page table에 매핑한다.
9. fault가 발생했던 명령어로 돌아가 실행을 계속한다.

이 흐름에서 SPT는 "이 주소의 page가 무엇인지"를 알려주는 중심 자료구조이고, frame table은 "실제 물리 메모리를 어떻게 나눠 쓸지"를 관리하며, swap table은 "메모리에서 밀려난 anonymous page를 어디에 저장할지"를 관리한다.

## 아직 더 확인해야 할 것

- stack growth를 허용할 정확한 조건
- `mmap`에서 eviction과 `munmap` 시 dirty page를 write-back하는 정확한 시점
- SPT hash table에서 중복 page 삽입을 막는 방식
- Project 2의 process loading, file descriptor, syscall 구현이 Project 3 VM과 충돌하지 않는지
- page fault handler에서 잘못된 user address와 stack growth 후보를 구분하는 기준

## 다음 학습/구현 방향

다음 단계에서는 개념을 코드 위치와 연결해야 한다.

- `vm/vm.c`: SPT 초기화, page 검색/삽입, frame 확보, page claim 흐름 확인
- `vm/uninit.c`: lazy loading initializer가 실제 page type으로 전환되는 흐름 확인
- `userprog/process.c`: `load_segment()`가 VM 방식으로 바뀌어야 하는 지점 확인
- `vm/anon.c`: anonymous page의 swap in/out 흐름 확인
- `vm/file.c`: mmap, file-backed page, dirty write-back 흐름 확인
- `include/lib/kernel/hash.h`, `lib/kernel/hash.c`: SPT 구현에 사용할 hash table API 사용법 확인

## 이번 기간의 핵심 성과

처음에는 VM을 "메모리를 더 크게 보이게 하는 기능" 정도로 이해했지만, 5월 8일부터 10일까지의 학습을 통해 VM이 `lazy loading`, `SPT`, `frame table`, `swap`, `mmap`, `dirty bit`, `hash table`이 함께 동작하는 구조라는 점을 이해하게 되었다.

특히 구현 관점에서는 `load_segment()`가 더 이상 파일을 즉시 메모리에 올리는 함수가 아니라, page fault 시점에 로드할 수 있도록 SPT에 정보를 등록하는 방향으로 바뀌어야 한다는 점을 배웠다. 이 점이 Project 2 방식과 Project 3 VM 방식의 가장 중요한 차이다.
